Devise, but for authorization.

The mechanism

Six core ideas

The whole engine is built from a handful of small, orthogonal concepts. Learn these six and you understand the entire model.

PPermission

A single controller#action pair. Permissions are auto-derived by walking your routes — you never hand-maintain a list.

reports#index
documents#approve

RRole

A named, editable bundle of permissions. Each subject has exactly one org-wide role.

Member = { projects#index,
  reports#index, reports#create… }

SScoped role

The same role definition, attached to one specific record. A subject can hold zero to many.

Reviewer of Report #42
Editor of Project X

★Owner

A role flagged full-access. Every permission granted, present and future — no per-action ticking.

Owner → full_access: true

≡Member

The baseline role every subject starts with. A normal editable bundle — nothing special about it structurally.

Member → baseline bundle

☠SoD veto

Separation of Duties (four-eyes): the subject who requested a thing cannot approve it. A non-configurable veto that overrides every role, including full-access.

the author of a document
cannot approve it

Positioning

Is this a known pattern?

Yes — but no single one. Grantwork is a deliberate composition of established access-control models. Here is where each piece sits.

RBAC
Role-Based Access Control. Permissions grouped into roles, roles assigned to subjects. Grantwork's org-wide role is textbook RBAC — but data-driven (roles live in the database and are edited in the UI, not hard-coded).
Hierarchical
RBAC
Roles inherit from parent roles. Grantwork deliberately avoids a hierarchy — a "manager" is expressed as a scoped role on the records they manage, not a rank that inherits everything below it.
ReBAC
Relationship-Based Access Control. Access depends on a relationship between subject and a specific resource ("Reviewer of this report"). Grantwork's scoped roles are exactly this.
ABAC / PBAC
Attribute / Policy-Based. Decisions from attributes and rules evaluated at request time. Grantwork borrows the flavour in one place only: the SoD veto is an attribute rule (requester ≠ approver) that runs before roles.

Grantwork = a composition, not a category

data-driven RBAC+ ReBAC scoped roles+ action-level permissions+ SoD veto

It also sits at a different layer than most Ruby authorization tools. An enforcement layer (like Pundit / ActionPolicy) asks "is this allowed?" but leaves you to store roles. A role-storage library (like rolify) stores roles but not action-level permissions or scoped resolution. An ability DSL (like CanCanCan) is code you write and maintain. An external PDP (like Cerbos / OpenFGA) is a separate service. Grantwork combines the pieces those keep separate — storage + derivation + scoped resolution + a managed UI — in one mountable engine.

Structure

Data model

Three relationships carry the whole system.

                         one org-wide role                bundle of permissions
   +-----------+   (belongs_to :role)   +-----------+   (has_many)    +---------------+
   |  SUBJECT  |----------------------->|   ROLE    |---------------->|  PERMISSION   |
   |  (a user) |          1 : 1         |           |     1 : many    | controller +  |
   +-----------+                        | full_     |                 | action        |
        |                               | access?   |                 | (auto-derived |
        |                               +-----------+                 |  from routes) |
        |                                     ^                        +---------------+
        | has_many :scoped_roles              |
        |                                     | (same role definition,
        v                                     |  reused when scoped)
   +------------------+                       |
   |  SCOPED_ROLE     |-----------------------+
   |                  |
   |  role_id ........+---> ROLE
   |  resource (polymorphic: resource_type + resource_id)
   |     e.g. Report #42, Project X, Document #7
   +------------------+
       0 .. many per subject

   Reading it:
     - A SUBJECT has exactly ONE org-wide ROLE.
     - A ROLE is a named set of PERMISSIONS (the grid).
     - A SUBJECT has ZERO-to-MANY SCOPED_ROLEs, each = (a ROLE) + (one polymorphic RESOURCE).

Permissions are rows in a join of role → permission; scoped roles reuse the very same role definitions, just pinned to a record.

Reference

What each right means

Because permissions are controller actions, granting them maps cleanly onto the seven RESTful actions — plus any custom action your controller defines.

REST actionHTTPGranting it means the subject can…
indexGET /reportsList the records
showGET /reports/:idView a single record
newGET /reports/newOpen the add form
createPOST /reportsAdd a new record
editGET /reports/:id/editOpen the edit form
updatePATCH /reports/:idEdit an existing record
destroyDELETE /reports/:idRemove a record
approve (custom)POST /reports/:id/approvePerform a custom action (subject to the SoD veto)

new+create and edit+update are usually ticked together — the grid presents them as one column pair.

The decision

How a request is resolved

For every controller#action, the resolver runs these steps in order and stops at the first that decides. It is fail-closed: no matching grant means DENY.

1

Separation of Duties veto DENY (wins over all)

Is the current subject the one who requested the thing being approved? If yes → DENY immediately. This beats everything below, including full-access. The author of a document cannot approve it.

if not vetoed ↓
2

Full-access role ALLOW

Does the subject's org-wide role have full_access? If yes → ALLOW (an Owner).

else ↓
3

Org-wide role has the permission ALLOW

Does the subject's one org-wide role include this controller#action? If yes → ALLOW.

else ↓
4

Scoped role on this record has it ALLOW

Does the subject hold a scoped role on the exact record being acted on whose bundle includes this action? If yes → ALLOW. (A scoped role on a different record does not count.)

else ↓
×

Default DENY (fail-closed)

Nothing granted it → DENY. The safe default is always "no".

Delivery

Scaffolded management UI

Like Devise ships views you can generate and customise, Grantwork's generators emit the management screens — mounted, styled, and yours to override.

rails g grantwork:install

Roles index

Lists every role, its permission count, and whether it is full-access. New / edit / delete from here.

rails g grantwork:views

Role editor (the grid)

The name + full-access toggle + auto-derived controller × action permission grid shown above.

rails g grantwork:views

Resource assignment

The subject page and the two attach-scoped-role screens — from a record, or from a subject.

Generated views are plain Rails templates; override any of them in your app just as you would Devise's.

Additional ideas & concepts

✎ Ideas · not yet decided

idea A. Feature flags (complementary layer)

A feature flag answers a different question than permissions: "is this capability even live for this actor?" — global, per-role, per-subject, or a % rollout. It would sit in front of the permission check, not replace it. Flag decides whether the feature exists; permissions then decide who may act; SoD still vetoes last.

1 · Feature flag
Is report_approvals live for this actor? (global / per-role / per-subject / % rollout)
↓ if live
2 · Permission check
May this actor do reports#approve? (org role or scoped role)
↓ if allowed
3 · SoD veto
Requester ≠ approver? (the author of a report can't approve it)
→ capability runs

    Open sub-questions

  • Where does the flag store live — inside the engine, or a separate flag gem it just reads?
  • If a flag is off, is the action hidden entirely or shown-but-blocked (and how does that differ from a permission "no")?

idea B. Ambient / portable authorization context

The same authorization context — subject = current_user + the resolver — flows from the controller down into views and ViewComponents without being threaded through by hand. A component just asks allowed_to?(:approve, report) / can? to hide a button the subject can't use, hitting the same engine as the controller gate. One authorization truth, top to bottom.

all three read ONE ambient context (subject + resolver) — no manual threading
