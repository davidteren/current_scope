## 2. The guidance: explicit passing vs CurrentAttributes vs dry-effects

The combined palkan / Evil Martians position is a ladder, not a ban:

1. **Explicit arguments are the default.** Props/kwargs everywhere you reasonably can (ViewComponent in the Wild I; Layered Design throughout â€” the final Chapter 4 refactoring removes `Current` from the model entirely and passes the actor as `post.destroy_by(user)`).
2. **`CurrentAttributes` is legitimate for request-scoped *metadata* set once at the boundary** â€” current user, tenant, locale, request id â€” provided it is written in the controller/job entry point and treated as read-only below (Layered Design Ch. 4; the book even shows an `around_action` setting locale/timezone as a *good* use). The book credits its two safety properties: per-execution-context isolation ("race conditions between different execution contexts are impossible") and automatic reset "at the end of every unit of work (a web request, background job, and so on)."
3. **dry-effects Reader when you need stronger discipline** â€” palkan calls it the "safer and predictable" way to move context across layers (Gemfile of dreams) because the handler block lexically delimits the scope and the required `include` documents exactly which objects depend on ambient state (Layered Design pp. 82â€“83; ViewComponent in the Wild II).

The book names `CurrentAttributes`' **three design flaws** (Google Books, Ch. 4, verbatim): "values can be written and read from anywhere, and no ceremony is required; reading unset values is possible (the result value would be nil); the same attribute can be written multiple times during the lifetime of the execution context."

### Why vanilla CurrentAttributes is still the right call for current_scope

- **The gem's use case is exactly the sanctioned one.** An authorization context (verified current user + resolver) is the textbook "request-scoped metadata set once at the boundary." The book's own authorization framing makes ambient context the authnâ†’authz seam: *"the responsibility of authentication is to populate the execution context with the current actor (verified), while the responsibility of authorization is to ensure that the actor has enough permissions... authorization validates the execution context"* (Layered Design Ch. 9). We are not passing business data through a global â€” we're holding the actor the framework verified.
- **The "Extended Rails Way" bar rules out dry-effects as a dependency.** palkan's own design standard is that a gem in this space should feel like a missing built-in Rails component ("tome" interview; Action Policy is his reference implementation). `CurrentAttributes` is the Rails-native primitive â€” Rails 8's built-in authentication generator itself ships a `Current` class. Pulling in dry-rb algebraic effects for an authorization engine fails the least-astonishment test for mainstream Rails apps and adds a dependency for what amounts to a stack-scoped variable.
- **The book's advice for dry-effects is to *emulate its properties*, not necessarily adopt it.** What makes the Reader effect safer is: declared access (you must `include` a behaviour to read), no silent nil (fails or falls back to an explicit default), and block-delimited writes. All three are implementable *on top of* vanilla `CurrentAttributes` (see Â§3). Action Policy itself demonstrates this: ambient at the boundary, explicit inside policies, with `through: -> { Current.user }` as a documented bridge (Authorization Context docs).
- **Precedent: Action Policy's view story is "ambient via the shared behaviour object" and nobody complains.** `helper_method :allowed_to?` means views already read authorization context ambiently through the controller instance (rails/controller.rb source). current_scope's `Current` extends the same reach to ViewComponents and jobs with one source of truth instead of a controller-only one.

**Design conclusion:** vanilla `CurrentAttributes` as the storage primitive, with the gem's public API engineered to close the three named flaws â€” write access confined to block-scoped boundary helpers, reads that distinguish "anonymous" from "unset," and policies/resolvers that take the actor as an explicit argument so the ambient layer is a convenience shell, never a hard dependency (mirrors the book's `def discard(by: Current.user)` compromise pattern, Ch. 4 p. 69).

## 4. API ideas worth adopting from Action Policy

1. **Two-sided context contract, fail-fast.** Policies/resolvers declare the context they need; the boundary declares how it's provided; a missing or nil key raises at construction with the key named (`Missing policy authorization context: user`). Even with ambient `Current`, keep the declaration so wiring drift is caught at the boundary, not deep in a rule. *(Authorization Context docs; issue #88 â€” the fix is always provider-side.)*
2. **Pluggable provider via one tiny hook.** `through:` accepts a method symbol or a proc `instance_exec`'d on the caller (`authorize :user, through: -> { @user || Current.user }`). One kwarg covers Devise, `CurrentAttributes`, ivars, and multi-tenant lookups. current_scope's resolver source should be exactly this shape. *(Authorization Context docs.)*
3. **Distinct nil semantics.** `allow_nil: true` (key present, value may be nil â€” guest/anonymous) vs `optional: true` (key may be absent). Conflating them loses "anonymous user" vs "forgot to wire the context." Maps directly onto the `user` / `user!` reader split. *(Authorization Context docs.)*
4. **Three-tier query API.** `authorize!` (raising guard, returns the record), `allowed_to?` (boolean for UI conditionals â€” current_scope's headline method), `allowance_to` (result object with failure reasons â€” powers API error payloads and disabled-button tooltips, useful for the separation-of-duties veto explaining *why*). *(Behaviour docs.)*
5. **One host-agnostic behaviour mixin.** A single module powers controllers, channels, services, and ViewComponents identically; class-level config lives in an inherited-and-dup'd map so subclasses extend without mutating parents. *(Behaviour docs; behaviour.rb source.)*
6. **Views via `helper_method` delegation, zero extra machinery.** Three `helper_method` lines make views call back into the controller's behaviour instance, inheriting context and memoized lookups for free. Don't build a parallel view-context system. *(rails/controller.rb source.)*
7. **ViewComponent integration in three lines** (production-proven): include the behaviour, `delegate :current_user, to: :helpers`, `authorize :user, through: :current_user` â€” with current_scope's `Current` as the fallback when helpers aren't available. *(passerelle.)*
8. **`verify_authorized` safety net.** An `after_action` that raises if no authorization call happened during the action, with class-level and inline skips â€” implemented as cheap counters on the behaviour object, not middleware. Directly applicable to auto-derived controller#action permissions. *(Rails integration docs.)*
9. **Convention-driven inference with overrides everywhere.** `current_user` assumed by convention; rule inferred from `action_name` (`authorize!` â†’ `:"#{action_name}?"`); resource-less `authorize!` infers the target from `controller_name`. Everything overridable per-class or app-wide. Matches current_scope's controller#action-derived permissions. *(Rails integration docs.)*
10. **Per-call context merge as the escape hatch.** `authorize! record, context: {account: other}` merges with the implicit context â€” override only what you need. Without this, impersonation and tenant-switching flows get painful. *(Authorization Context docs.)*
11. **Immutable context snapshots.** Caller-side context is lazily built and memoized per instance (= per request); policy-side context is validated, exposed as attr_readers, and **frozen at construction** â€” which is what makes caching and test assertions safe. Freeze current_scope's resolved context the same way. *(behaviour.rb source.)*
12. **Testing story.** (a) Policies test as POROs with context as constructor kwargs â€” no framework needed; (b) opt-in RSpec DSL (`describe_rule`/`succeed`/`failed` with `let(:context)`); (c) matchers for the *act* of authorization (`be_authorized_to(...).with_context(...)`, `have_authorized_scope`) with subset-based context matching so tests assert only the keys they care about; (d) per-thread cache off in test env by default. *(Testing docs; thread_memoized.rb.)*
13. **Scoping-based authorization as a first-class feature.** `authorized_scope`/`relation_scope` replaces find-then-check and mitigates N+1 on collections â€” a natural fit for current_scope's scoped per-record roles. *(Chapter09 examples; Layered Design Ch. 9 p. 189.)*
